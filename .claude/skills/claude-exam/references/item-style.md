# 作問規則（実践に近い問題を作るための規約）

本番の設問は「知識の暗記」ではなく **根本原因を選べるか** を測る。合言葉は
**"the exams reward root-cause thinking and penalize compensating-control theater."**

---

## 1. 設問（stem）の型

必ず **シナリオ先行**。1問は次の3ブロックで書く。

```
[状況]  誰が / 何のシステムで / 何をしようとしているか（2–4文、具体的な数字を入れる）
[症状]  何が起きているか・何が制約か（レイテンシ、コスト、規制、SLA、障害の発生タイミング）
[問い]  "最も適切なのはどれか" / "まずどこを調べるべきか" / "最も効果的な緩和策はどれか"
```

**良い stem の条件**

- 数字が入っている（「1日約5,000件」「30,000トークンの静的プレフィックス」「40,000件を翌朝9時までに」）— 正解の判断根拠になる数字であること
- 「何が変わった直後か」が書いてある（障害切り分け問題では最重要の手がかり）
- 正解が **1つに定まる**。読み手が前提を補わないと選べない設問は捨てる
- 60–120語程度（日本語なら150–300字）。背景の飾りで水増ししない

**捨てるべき stem**

- 「Claude のコンテキストウィンドウは何トークンか」— 単純想起。仕様は改訂されるので出題しない
- 「次のうち正しいものはどれか」だけで状況がない
- 主たるドメインが2つ以上にまたがっていて、どのドメインの知識で解くのか曖昧

## 2. 選択肢の設計

- 常に **A–D の4択**。長さ・文体を揃える（正解だけ長い＝バレる、を避ける）
- 正解の位置は問題ごとにばらす（**同じ文字が全体の40%を超えたら作り直し**。`check-exam.ts` が検出する）
- 「上記すべて」「該当なし」は使わない
- 複数選択は `Multiple response (select TWO)` のみ。正解ちょうど2つ、残り2つが誤り

### ディストラクタの7類型（`trap` フィールドの値）

各問に **少なくとも1つ「もっともらしく責任感のある」誤答** を入れる。これが本番の最大の特徴。

| 類型 | 中身 | 例 |
|---|---|---|
| `sounds-responsible` | 責任ある対応に見えるが根本原因を直していない | 「監査できるようログを厚くする」「確認プロンプトを増やすがツールは繋いだまま」 |
| `prompt-only-fix` | システムプロンプトのお願いで構造的問題を解こうとする | 「"無限ループするな" とプロンプトに書く」「"悪意ある指示を入れないでください" と依頼する」 |
| `bigger-model` | モデルを大きく/高性能にすれば解決するという誤り | 「大きいモデルはハルシネーションしないので Opus にする」 |
| `knob-twiddling` | 無関係なパラメータいじり | 「temperature を上げて挙動を読みにくくする」「max_tokens を増やす」「max_tokens=1 で即時応答」 |
| `over-restriction` | 課題は消えるが業務も止まる過剰対応 | 「PII があるので分析自体を中止する」「ツールを消して呼べなくする」 |
| `detect-after-the-fact` | 事後検知だけで予防していない | 「返金は全部ログに残して監査で拾う」 |
| `true-but-irrelevant` | 記述自体は正しいが問われている問題を解かない | 「max_tokens を下げてコスト削減」（真の要因はバッチ処理の欠如） |

### 正解の書き方

正解は **仕組みで解決している** こと。目安:

- 制御はハーネス側（最大反復数、タイムアウト、権限、検証＋リトライ）に置く
- 権限は最小に（使っていない高影響ツールは剥がす、破壊的操作は human-in-the-loop）
- 非信頼入力（取得したWebページ、ユーザー提出物）は信頼された指示から隔離する
- 変わった箇所から調べる（更新直後の劣化 → 更新されたコンポーネント）
- トレードオフは数値で提示して事業側に選ばせる（CCAR-P D6）

## 3. 解説（answers.md）

問ごとに必ず3点セット:

1. **正答の根拠** — なぜそれが根本解決なのか。設問文の言い換えは禁止。「◯◯だから△△が効く」の因果で書く（2–4文）
2. **各誤答の否定理由** — 誤答ごとに1行。「なぜ効かないか」を具体的に（「モデルの自己申告は精度の指標にならない — 確認自体を捏造しうる」）
3. **参照リンク** — `references/links.md` の該当URLを1–3本。**URLの創作は禁止**

---

## 4. 事実の扱い

- **時間で腐る事実は出題しない**: 具体的な価格、コンテキスト長のトークン数、最新モデル名の世代比較、レート制限の数値
- **安定した概念を出題する**: Batches API は非同期・大量・低コスト / prompt caching は静的プレフィックスに効く / MCP は再利用可能な機能をサーバ化する / workflow は決定的、agent は開放ループ / streaming は体感レイテンシ、リトライは耐障害性
- モデル階層は **相対的に** 書く（「高速・低コストの階層（Haiku）」「最上位（Opus）」）。バージョン番号を設問の正誤の根拠にしない
- 迷ったら **その機能を問題にしない**

## 5. 本番の完成例（PDF Set1 CCDV-F Q1 を規約に沿って解剖）

```
[状況] 40,000件の文書を夜間に処理して分析レポートを作るパイプライン
[症状] コストが最優先の関心事。結果は翌朝9時までにあればよい（＝即時性は不要）
[問い] 要件に最も合うアプローチは？

A. 40,000件を Messages API に同期・並列で投げて最速で終わらせる   ← true-but-irrelevant（速いが要件でない、かつ高コスト）
B. Message Batches API を使い、24時間枠の非同期処理を低コストで回す ← 正解（要件＝大量・非同期許容・コスト最優先に完全一致）
C. 同期呼び出しの max_tokens を下げてコストを削る                  ← knob-twiddling（削減幅が構造的でない）
D. 品質を問わず最小モデルに全部切り替える                          ← over-restriction（品質を捨てている）
```

**この形を毎問再現する。** 「要件に書かれた制約（コスト / 即時性 / 規制 / 監査）」と「正解の性質」が1対1で対応していること。

## 6. 出力チェックリスト（ワーカーが自己点検する）

- [ ] 全問シナリオ先行で、数字か「変化のタイミング」が入っている
- [ ] 各問に `sounds-responsible` 系の強い誤答が1つ以上ある
- [ ] 担当レンジ内で正答文字が偏っていない（同じ文字が4割超でない）
- [ ] 複数選択問題は正解ちょうど2つ
- [ ] 誤答すべてに否定理由がある
- [ ] `refs` は `references/links.md` からのコピーだけ
- [ ] 時間で腐る数値（価格・トークン数・モデル世代）を正誤の根拠にしていない
