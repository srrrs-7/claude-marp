# スライド設計ルール（行動心理学 + Google ベストプラクティス）

スライドデータJSON生成時・マークダウン編集時に以下のルールを **必ず** 適用する。

---

## 0. 図解の必須ルール（最重要）

**すべてのプレゼンテーションで図解を必ず含める。テキストのみのスライドが3枚以上連続してはならない。**

### 図解の挿入基準

- **2〜3スライドごとに最低1つ** のビジュアル要素（SVG図・コードブロック・表・アイコン的記号）を挿入する
- プロセス・フローの説明 → **必ずSVGで図解**
- システム構成・アーキテクチャ → **必ずSVGで図解**
- 時系列・手順 → **必ずSVGで図解**
- 比較・割合 → **必ずSVG or 表（テーブル）で図解**
- 関係性・依存 → **必ずSVGで図解**
- データ・統計 → **必ず表またはSVGグラフで図解**
- 概念の説明 → **テキストだけで済ませず、SVGで構造を可視化**

### 図解はSVGを使用する（必須）

**デフォルト: SVGを使用する。Mermaidは使用しない。**

すべての図解はインラインSVGで作成する。SVGはMermaidと比べて以下の利点がある:

| 判断基準 | SVG | Mermaid |
|---------|-----|---------|
| **視覚品質** | ◎（色・レイアウト・凡例を完全制御） | △（自動レイアウト依存） |
| **デザイン自由度** | ◎（任意の図形・配色・矢印スタイル） | △（定型パターンのみ） |
| **プレゼン適性** | ◎（プロフェッショナルな見た目） | △（テキスト的な見た目） |
| **レンダリング安定性** | ◎（どの環境でも同一表示） | △（エンジン依存で崩れうる） |
| **サイズ制御** | ◎（viewBox + max-height で確実に収まる） | △（はみ出しリスクあり） |

**SVG作成ルール:**

1. **インラインSVGとしてMarkdownに直接埋め込む**（外部ファイル参照ではなく）
2. **`viewBox` を必ず指定** — レスポンシブに拡縮するため
3. **`style="max-height:70vh;width:auto;display:block;margin:0 auto;"` を付与** — スライド枠内に収める
4. **色分けでコンポーネントの役割を区別** — 凡例をSVG内に含める
5. **ドロップシャドウ (`filter: feDropShadow`) で立体感** — 視認性を高める
6. **矢印マーカー (`<marker>`) でフロー方向を明示** — 実線/破線で種別を区別
7. **日本語テキストは `font-family="sans-serif"` を指定** — フォント互換性を確保

**図の種類別テンプレート:**

| 図の種類 | SVG構成要素 |
|---------|-----------|
| フロー図 | rect(角丸) + line(矢印marker) + text |
| アーキテクチャ図 | rect(色分け) + line(実線/破線) + 凡例 |
| シーケンス図 | rect(参加者) + line(矢印) + text(メッセージ) |
| 比較図 | rect(並列配置) + text + 色分けで対比 |
| ベン図 | circle(半透明) + text(ラベル) |
| タイムライン | line(横軸) + circle(イベント) + text |

### テキスト→ビジュアル置換の原則

箇条書きで5つ以上の項目を並べるなら、それは図にすべきサイン。以下の変換を検討する:

| テキスト表現 | ビジュアル変換先 |
|---|---|
| 手順の列挙 | SVGフロー図（rect + arrow） |
| 比較リスト | 2カラム表 or SVG対比図 |
| 構成要素の説明 | SVG構成図（rect + line） |
| 数値の羅列 | SVGグラフ or 表 |
| 時系列イベント | SVGタイムライン |
| カテゴリ分類 | SVGグルーピング図 |

## 1. 認知負荷の管理（Cognitive Load Theory）

人間のワーキングメモリは一度に **3〜5チャンク** しか処理できない。

### 1スライド1メッセージ

- 1枚のスライドで伝えるメッセージは **1つだけ**
- 箇条書きは **最大3つ**（理想）、絶対に **4つ以下**。超える場合は2枚に分割するか図解に変換

### テキスト量の制限（5-5-5ルール + Google推奨）

- **5行以下**: 1スライドのテキスト行数
- **5語（日本語20文字）以下/行**: 1行あたりの語数
- **フォント30pt以上**: 遠くからでも読める最小サイズ（Guy Kawasaki 10-20-30ルール）
- 読み上げ原稿をスライドに書かない（冗長性効果の回避）

### 段階的開示

- 複雑な図やプロセスは一度に見せず、ステップごとに複数スライドに分割
- スピーカーノートはスライド内容の「補足・展開」にする

## 2. 構成と記憶効果

### 10-20-30ルール（Guy Kawasaki）

- **10枚以内**: 1つのプレゼンの理想スライド数
- **20分以内**: プレゼン時間の上限目安
- **30pt以上**: フォントの最小サイズ

### 系列位置効果（Serial Position Effect）

人は最初（初頭効果）と最後（新近効果）に提示された情報を最もよく記憶する。

- **冒頭3枚**: 最も重要なメッセージ・問題提起。「なぜこの話を聞くべきか」を最初に
- **中盤**: 詳細・根拠・データ。記憶に残りにくいため **図解で視覚的に補強（必須）**
- **終盤3枚**: キーテイクアウェイ・行動喚起。聴衆が持ち帰るメッセージを明確に

### ピークエンドの法則（Peak-End Rule）

人は最も印象的な瞬間（ピーク）と最後の印象で体験を評価する。

- **ピークスライド**: プレゼン中盤にインパクトのある1枚を意図的に配置（驚きのデータ、ビフォーアフター図、デモ結果）
- **エンドスライド**: 単なる箇条書きで終わらせない。聴衆への問いかけ・行動喚起・印象的な一文で締める

### 3の法則（Rule of Three）

- **キーメッセージは3つ**: メインの主張・論点は3つに絞る
- **セクション構成**: 「問題→解決→効果」「What→Why→How」など3部構成
- **箇条書きの理想数**: 3つが最も記憶に残る

## 3. ビジュアルファースト設計（Google推奨）

### 画像優位性効果（Picture Superiority Effect）

テキストのみの情報は3日後に10%しか記憶されないが、画像と組み合わせると **65%** に上昇する。

### ビジュアル要素の使い分け

| 要素 | Marpでの実現方法 | 使用場面 |
|---|---|---|
| フロー図 | インラインSVG（rect + marker矢印） | プロセス、ワークフロー、意思決定 |
| シーケンス図 | インラインSVG（rect + line + text） | API通信、ユーザー操作フロー |
| 円グラフ | インラインSVG（circle + path） | 割合、構成比 |
| 表（テーブル） | Markdownテーブル（`content`内） | 比較、スペック、対比 |
| コードブロック | `code` + `codeLanguage` | 実装例、設定例（10行以内） |
| タイムライン | インラインSVG（line + circle + text） | 時系列、ロードマップ |
| 構成図 | インラインSVG（rect + line + 凡例） | アーキテクチャ、システム構成 |

### 統一されたビジュアル言語（Google推奨）

- **1プレゼン内でアイコンスタイル・色・形を統一**: 途中でデザインを変えない
- **SVG図の配色・スタイルも統一**: 全図で同じカラーパレット・角丸・フォントを使う
- **フォントは2種類まで**: 見出し用1つ + 本文用1つ

## 4. 視覚設計と注意誘導

### フォン・レストルフ効果（Von Restorff Effect）

均一な中で「異質なもの」は圧倒的に記憶に残る。

- **強調は1スライド1箇所**: 太字・色変え・サイズ変更は最も重要な1要素にだけ
- **全部強調 = 強調なし**: 複数の強調が競合すると効果消失
- **キースライド**: 通常と明確に異なるデザイン（背景色反転、大きなフォント1文のみ）を要所に配置

### ゲシュタルト原則

- **近接**: 関連する要素は近くに、無関係な要素は離す
- **類似**: 同じカテゴリの情報は同じ色・形・サイズで表現
- **余白**: スライドの30%以上を余白にする。余白は視線を誘導し理解を助ける設計要素

### 整列（Google推奨）

- すべてのテキスト・画像・図を **グリッドに沿って整列**
- ずれた要素はプロフェッショナルさを損なう

## 5. 色彩心理学

### 60-30-10 ルール

- **60%**: ベースカラー（背景・本文）— 視覚的安定
- **30%**: セカンダリカラー（見出し・枠線）— 構造の明示
- **10%**: アクセントカラー（強調・CTA）— 注意誘導

### 色の心理的効果

| 色 | 心理効果 | 適した用途 |
|----|----------|------------|
| 青 | 信頼・安定・冷静 | 企業プレゼン、データ分析、技術説明 |
| 緑 | 成長・安全・肯定 | 成果報告、環境、ヘルスケア |
| 赤 | 緊急・重要・警告 | 問題提起、リスク強調、CTA（控えめに） |
| オレンジ | 活力・親しみ・創造 | ワークショップ、アイデア発表 |
| 紫 | 革新・高級・創造性 | 新技術、ブランディング |
| 黒/濃灰 | 権威・洗練・プロ | テーマ `gaia` の `invert` モード |

### 暖色 vs 寒色の使い分け

- **暖色（赤・オレンジ）**: 緊急性・問題提起・CTA・警告に
- **寒色（青・緑）**: 分析的思考・信頼感・データ・解決策に
- **1プレゼン内で両方使う**: 問題（暖色）→ 解決（寒色）の流れで感情を設計

### コントラスト（Google推奨）

- 背景と文字のコントラストを十分に確保（ライト背景×ダーク文字 or ダーク背景×ライト文字）
- 衝突する色（黄色×白など）は避ける

## 6. 行動経済学の応用

### 損失回避（Loss Aversion）

人は「得をする喜び」より「損をする恐怖」に2倍強く反応する。

- **問題提起スライド**: 「〜しないと何が起こるか」を先に提示してから解決策を示す
- **フレーミング**: 「生産性が50%向上」より「年間1000時間のムダを削減」の方が響く
- **ビフォーアフター**: 現状の痛み → 導入後の改善を **対比図（2カラム or フロー図）** で見せる

### アンカリング効果（Anchoring Effect）

- **大きな数字を先に**: 全体コストの大きさを先に示してから削減額を提示
- **比較の基準**: 競合や業界平均を先に出してから自社の数字を見せると差が際立つ

### 社会的証明（Social Proof）

- **導入実績・事例**: 「Fortune 500の30%が採用」「GitHub Stars 50k+」
- **引用・推薦**: 業界の著名人やユーザーの声
- **データの視覚化**: 採用率の推移グラフ、満足度の数字を **必ず図表で表現**

## 7. 出典URLの明記（必須）

スライド本文・スピーカーノート・JSONデータ作成時に、参照した文献やデータの **URLを必ず記載** する。

### 必須要件

- 統計・調査結果・ベンチマーク・外部主張を使う場合、対応する出典URLを明記する
- URLは可能な限り一次情報（公式ドキュメント、論文、原著記事）を優先する
- URLが長い場合でも省略せず、完全なURLを残す
- URLが存在しない情報は「事実」として断定しない

### 記載方法

- Markdown (`docs/**/*.md`): 末尾に `## 参考文献` セクションを作り、箇条書きでURLを列挙
- JSON (`docs/**/*.json`): 該当スライドの `speakerNotes` に `出典: <URL>` 形式で記載

## 7. Marp固有の実践ルール

### レイアウト使い分け

```
layout: "center"   → タイトル、まとめ、インパクトスライド（1文のみ）
layout: "section"  → セクション区切り（新しい話題の開始を明示）
layout: "default"  → 通常コンテンツ
```

### テーマ選択ガイド

| テーマ | 色調 | 適したシーン |
|--------|------|-------------|
| `gaia` | カラフル・力強い | 技術カンファレンス、勉強会、LT |
| `gaia` + `invert` | ダーク | コード中心のプレゼン、夜間イベント |
| `default` | 白背景・落ち着き | 社内報告、ビジネス提案、顧客向け |
| `uncover` | ミニマル | デザイン系、アカデミック、ワークショップ |

### スライド枚数の目安

| 発表時間 | 推奨枚数 | 1枚あたり |
|----------|----------|-----------|
| 5分 LT | 5-7枚 | 約45秒 |
| 15分 | 10-12枚 | 約80秒 |
| 20分 | 12-18枚 | 約80秒 |
| 45分 | 25-35枚 | 約90秒 |

15〜20枚が最も効果的な範囲（認知負荷の観点）。50枚超は情報過多。

### スピーカーノートの書き方

- スライドの箇条書きをそのまま繰り返さない（冗長性効果）
- 「なぜこれが重要か」「聴衆への問いかけ」「具体例・エピソード」を書く
- トランジション（次のスライドへの橋渡し文）を含める
- 1スライドあたり3〜5文を目安

### アニメーション・トランジション（Google推奨）

- トランジションは **控えめに**。注意を引くためではなく、情報の流れを助けるために使う
- フェードインで要点を段階的に表示するのは有効
- 派手なエフェクトは聴衆の集中を妨げる

## 8. コンテンツ量の制約（スライド枠はみ出し防止）

### コードブロックの制約

**コードブロックはスライド高さを大きく消費する。以下を厳守:**

- **推奨最大行数: 8行** — これを超えると箇条書きとの共存が困難
- **絶対上限: 12行** — これを超えると確実にはみ出す（Gaiaテーマ）
- **12行を超える場合の対応**:
  - コードを短縮する（不要な行を削除、コメント削減）
  - 2スライドに分割する
  - 重要部分のみ抜粋し、スピーカーノートに完全版を記載

### コードブロック併用時の箇条書き制約

**コードブロックとテキストが同居する場合、箇条書きを抑制:**

| コード行数 | 箇条書き最大数 | 理由 |
|-----------|--------------|------|
| 0行（コードなし） | 4項目 | テキストのみで認知負荷が高い |
| 1-6行 | 3項目 | コードとテキストのバランス |
| 7-10行 | 2項目 | コードが主体、テキストは補足 |
| 11-12行 | 1項目 | コードのみに集中させる |

### SVG図の制約

- **複雑な図（ノード8個以上）は1スライドに1つのみ**
- **図 + 箇条書き3項目を超える場合は分割**
- **`max-height: 70vh` を必ず指定**: スライド枠内に収めるため
- **図の縦幅を意識**: シーケンス図・タイムラインは縦に伸びやすいため注意

### 1スライドの総コンテンツ上限（Gaiaテーマ・16:9）

**有効高さ約510px、1行約30-40pxとして算出:**

- タイトル (h1): 約80px
- 箇条書き1項目: 約40px
- コードブロック全体: `行数 × 21px + 40px`（パディング含む）
- SVG図: 約200-300px（複雑度による、max-height: 70vhで制御）

**はみ出し判定式:**
```
総高さ = 80 (タイトル) + (箇条書き数 × 40) + (コード行数 × 21 + 40) + 図の高さ
総高さ > 510px ならはみ出し確定 → 分割必須
```

### スライド生成時のチェックリスト

スライドデータJSON作成時、以下を確認:

- [ ] コードブロックは8行以内か？（推奨）
- [ ] コードブロック+箇条書きの組み合わせは上記表に従っているか？
- [ ] コード12行を超える場合、2スライドに分割したか？
- [ ] SVG図が複雑な場合、箇条書きを削減したか？
- [ ] タイトル+コンテンツが510px以内に収まるか？

**違反例（7ページ目の問題）:**
```json
{
  "title": "...",               // 80px
  "content": ["...", "...", "..."],  // 3項目 = 120px
  "code": "12行のコード",        // 約292px
  "layout": "default"
}
// 合計: 80 + 120 + 292 = 492px → ギリギリだが、実際はフッター・余白でオーバー
```

**修正例:**
```json
{
  "title": "...",
  "content": ["...", "..."],    // 2項目に削減
  "code": "8行に短縮",
  "layout": "default"
}
```
