# 証明書の構成要素：Subject・SAN・有効期限・署名アルゴリズム

> *SANのURIフィールドにSPIFFE IDを埋め込むことでワークロード識別を標準化できる*

- Subject（主体者識別）: CN=auth-service, O=MyOrg, C=JP
- Subject Alternative Name（SAN）: DNS:auth.internal, URI:spiffe://cluster.local/ns/default/sa/auth
- Issuer（発行者）: CN=Intermediate CA, O=MyOrg
- 有効期限: NotBefore / NotAfter（短命証明書は24〜90日推奨）
- 公開鍵アルゴリズム: ECDSA P-256（RSA-2048より高速・短い鍵長）
- 署名アルゴリズム: SHA-256 with ECDSA
- 拡張フィールド: KeyUsage（digitalSignature）, ExtendedKeyUsage（serverAuth / clientAuth）

# EnvoyプロキシでmTLSを設定する：実装はDownstreamとUpstreamを分けて考える

- Envoyのmtls設定は listener（受信）と cluster（送信）の2箇所に記述する
- transport_socket に DownstreamTlsContext / UpstreamTlsContext を使い分ける
- require_client_certificate: true で相互認証を強制（省略すると片方向TLSになる）
- validation_context で信頼するCA証明書を指定し、不正なクライアント証明書を拒否する


---

# EnvoyプロキシでmTLSを設定する：実装はDownstreamとUpstreamを分けて考える（コード例）

```yaml
# Envoy mTLS 設定サンプル (envoy.yaml)
static_resources:
  listeners:
  - name: listener_mtls
    address:
      socket_address: { address: 0.0.0.0, port_value: 8443 }
    filter_chains:
    - transport_socket:
        name: envoy.transport_sockets.tls
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.transport_sockets.tls.v3.DownstreamTlsContext
          require_client_certificate: true   # ← mTLS必須
          common_tls_context:
            tls_certificates:
            - certificate_chain: { filename: /etc/certs/server.crt }
              private_key:       { filename: /etc/certs/server.key }
            validation_context:
              trusted_ca: { filename: /etc/certs/ca.crt }  # クライアント証明書検証用CA

  clusters:
  - name: upstream_service
    transport_socket:
      name: envoy.transport_sockets.tls
      typed_config:
        "@type": type.googleapis.com/envoy.extensions.transport_sockets.tls.v3.UpstreamTlsContext
        common_tls_context:
          tls_certificates:
          - certificate_chain: { filename: /etc/certs/client.crt }
            private_key:       { filename: /etc/certs/client.key }
          validation_context:
            trusted_ca: { filename: /etc/certs/ca.crt }   # サーバ証明書検証用CA
```

# Vault PKI SecretsエンジンでmTLS証明書を自動発行する：運用コストを90%削減する設計

- Vault PKI Secrets Engineは証明書の発行・失効・TTL管理をAPIで完全自動化する
- Roleで許可するドメイン・有効期間・鍵サイズを制限し、最小権限の原則を証明書発行に適用する
- pki_int（中間CA）を分離することでRoot CAをオフラインに保ち、侵害リスクを最小化できる


---

# Vault PKI SecretsエンジンでmTLS証明書を自動発行する：運用コストを90%削減する設計（コード例）

```bash
# Vault PKI Secrets Engine セットアップ
# 1. PKIエンジン有効化（Root CA と 中間CA を分離）
vault secrets enable -path=pki        pki   # Root CA
vault secrets enable -path=pki_int    pki   # 中間CA（サービス証明書発行用）

vault secrets tune -max-lease-ttl=87600h pki      # Root CA: 10年
vault secrets tune -max-lease-ttl=43800h pki_int  # 中間CA: 5年

# 2. Root CA 生成
vault write pki/root/generate/internal \
  common_name="MyOrg Root CA" ttl="87600h"

# 3. 中間CA CSR生成 → Root で署名
vault write -format=json pki_int/intermediate/generate/internal \
  common_name="MyOrg Intermediate CA" | jq -r '.data.csr' > pki_int.csr

vault write pki/root/sign-intermediate csr=@pki_int.csr \
  format=pem_bundle ttl="43800h" | jq -r '.data.certificate' > signed_int.pem

vault write pki_int/intermediate/set-signed certificate=@signed_int.pem

# 4. Role定義（サービス証明書の制約を設定）
vault write pki_int/roles/microservices \
  allowed_domains="svc.cluster.local" \
  allow_subdomains=true \
  max_ttl="24h"   # 短命証明書で失効管理不要

# 5. 証明書発行（サービスが起動時に自動実行）
vault write pki_int/issue/microservices \
  common_name="payment.svc.cluster.local" \
  ttl="24h"
```

# 証明書管理でよくある失敗パターンと、それを防ぐアーキテクチャ的回避策

> *障害の90%は「気づかなかった」ではなく「自動化していなかった」が原因*

- 【失敗1】証明書の有効期限をカレンダーで管理 → 更新漏れでサービス停止（週末深夜の緊急対応）
- 【失敗2】Root CAの秘密鍵をサーバ上に平文保存 → 侵害時に全証明書が無効化必須
- 【失敗3】全サービスで同一証明書を使い回し → 1枚の漏洩で全サービスへの横断攻撃が可能
- 【失敗4】証明書の更新をDeploymentと同じパイプラインで管理 → ローリングアップデート中に旧/新証明書が混在
- 【失敗5】OCSPレスポンダへのネットワーク経路を考慮せず → CAアクセス不能時に全mTLS接続が失敗
- 【対策】短命証明書（TTL<24h）+ SPIRE自動発行 + Vault PKI(Root CA隔離) + cert-manager(K8s) + Prometheusアラート(cert_expiry_seconds<7d)

-->

# mTLSが過剰な場合はJWT・API Keyで十分なセキュリティを実現できる（1/2）

> *認証方式の選択は「誰が誰に」「どの環境で」通信するかで決まる*

- **判断基準: mTLS vs JWT vs API Key**
- mTLS採用条件: サービス間通信 + ゼロトラスト必須 + 証明書管理インフラあり
- JWT採用条件: ユーザー認証が主体 / 短命トークン / ステートレスAPIが必要
- API Key採用条件: 外部パートナー連携 / 管理が簡単 / セキュリティ要件が低め

# mTLSが過剰な場合はJWT・API Keyで十分なセキュリティを実現できる（2/2）

> *認証方式の選択は「誰が誰に」「どの環境で」通信するかで決まる*

- **mTLS の強み**: 秘密鍵が通信路に流れない・証明書失効で即座にアクセス遮断
- **JWT の強み**: インフラ不要・クロスドメイン対応・有効期限を柔軟に設定可能
- **API Key の弱点**: 漏洩リスク大・失効に伝搬遅延・通信経路を保護しない
- **推奨構成**: 内部サービス間=mTLS、外部API=JWT+mTLS（多層防御）

# gRPCでmTLSを有効化するGoコードは20行以内で実装できる（コード例）

```go
package main

import (
	"crypto/tls"
	"crypto/x509"
	"os"

	"google.golang.org/grpc"
	"google.golang.org/grpc/credentials"
)

// mTLS サーバー設定
func newServerCreds(certFile, keyFile, caFile string) (credentials.TransportCredentials, error) {
	cert, err := tls.LoadX509KeyPair(certFile, keyFile)
	if err != nil {
		return nil, err
	}
	caBytes, err := os.ReadFile(caFile)
	if err != nil {
		return nil, err
	}
	caPool := x509.NewCertPool()
	caPool.AppendCertsFromPEM(caBytes)

	tlsCfg := &tls.Config{
		Certificates: []tls.Certificate{cert},
		ClientAuth:   tls.RequireAndVerifyClientCert, // mTLS必須
		ClientCAs:    caPool,
		MinVersion:   tls.VersionTLS13,
	}
	return credentials.NewTLS(tlsCfg), nil
}

// mTLS クライアント設定
func newClientCreds(certFile, keyFile, caFile string) (credentials.TransportCredentials, error) {
	cert, err := tls.LoadX509KeyPair(certFile, keyFile)
	if err != nil {
		return nil, err
	}
	caBytes, _ := os.ReadFile(caFile)
	caPool := x509.NewCertPool()
	caPool.AppendCertsFromPEM(caBytes)

	return credentials.NewTLS(&tls.Config{
		Certificates: []tls.Certificate{cert},
		RootCAs:      caPool,
		MinVersion:   tls.VersionTLS13,
	}), nil
}

func main() {
	serverCreds, _ := newServerCreds("server.crt", "server.key", "ca.crt")
	grpcServer := grpc.NewServer(grpc.Creds(serverCreds))
	_ = grpcServer // RegisterServiceServer(grpcServer, &MyService{})

	clientCreds, _ := newClientCreds("client.crt", "client.key", "ca.crt")
	conn, _ := grpc.NewClient("svc:443", grpc.WithTransportCredentials(clientCreds))
	_ = conn
}
```
