# CLAUDE CERTIFICATION PROGRAM

## Claude Certified Architect – Foundations (CCA-F)

**30 items | 120 minutes | $125 | 720/1000 to pass**（本番は 60 問・全問シナリオベース。別称 CCAR-F）

Blueprint: D1 Agentic Architecture & Orchestration 27% ・ D2 Claude Code Configuration & Workflows 20% ・ D3 Prompt Engineering & Structured Output 20% ・ D4 Tool Design & MCP Integration 18% ・ D5 Context Management & Reliability 15%

### How to use this set

- **時間を計って解く。** 30問。1問あたり約2分が本番のペース配分です
- **解答は別冊（`answers.md`）にあります。** 全問解き終えるまで開かないでください
- **正解の「文字」ではなく根拠を覚える。** 本番では選択肢の順序も文言も変わります。なぜ正しいのかを言語化できるまで `answers.md` の解説を読み込んでください
- **間違えた問題はドメイン別に記録する。** 偏って落としているドメインの公式ドキュメントに戻るのが最短です
- **複数選択（Multiple response）は部分点なし。** 2つとも正しく選べて初めて正答です

D. 判定を出力するモデルを上位階層に切り替え、あわせて temperature を下げて出力のばらつきを抑える

### Q30 Single response

**D5 — CONTEXT MANAGEMENT & RELIABILITY**

1回の実行で900件を処理する過程で、`get_vendor_status` と `fetch_weather_alerts` の生の応答（1件あたり数十行のJSONと自由記述の警報テキスト）がすべて会話履歴に積み上がっている。300件を超えたあたりから、エージェントは直前に確認したばかりの在庫数を取り違え、後半では初期に処理した出荷の判定を根拠なく蒸し返すようになる。最も適切な対処はどれか。

A. 会話履歴が一定の長さに達したら、それまでの全件の生応答をエージェント自身に要約させ、要約だけを残して以降の判定を続けさせる
B. `get_vendor_status` と `fetch_weather_alerts` の応答から自由記述の警報テキストを除去し、数値フィールドだけを履歴に残して1件あたりの分量を圧縮する
C. エージェントの実行を1回あたり300件までに分割し、301件目以降は別の実行として翌日に回すことで、履歴が劣化する水準に達しないようにする
D. 件ごとの生応答を実行の外部ストアに保存し、エージェントの文脈には判定に必要な要約フィールドと参照キーだけを載せ、詳細が要る件だけ参照キーで取り出させる
