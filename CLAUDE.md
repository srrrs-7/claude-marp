# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Bun 1.3.5 + TypeScript CLI: Zod-validated `slides-data.json` + `slides.config.yaml` → Marp `.md` → HTML/PDF/PPTX via `bunx @marp-team/marp-cli`. No AI SDKs — slide content is generated by Claude Code / Codex. `docs/` 全体（233+ デッキ）が GitHub Pages として公開される（`main` への push で `deploy-pages.yml` が `generate:index` → deploy）。

**詳細ルールは path-scoped rules（`.claude/rules/` — 末尾の一覧参照）に分割済み。** 該当ファイルを触ると自動ロードされる。本ファイルは毎セッション全文ロードされるため**常時必要な方針のみ**を置き、詳細の追記は対応する rule に行う（本ファイルは 200 行以下を維持）。

## 🔒 Security — bun を host で実行することを全面禁止（厳格ルール）

**host での `bun` / `bunx`（および npm / npx / yarn / pnpm）の実行は禁止。`bun install` は絶対に host で実行しない。** すべての bun runtime 作業（install / format / typecheck / test / render / export / validate / stats）は devcontainer コンテナ内で実行する。

- **理由: Shai-Hulud 型 npm サプライチェーン攻撃対策。** install スクリプトが host の認証情報（`~/.ssh`・キーチェーン・クラウド credential）に触れる経路を遮断し、影響範囲をコンテナ内に限定する
- **強制: `.claude/hooks/pre-bash-guard.sh`（PreToolUse）が host での bun 系コマンドを exit 2 でブロックする。** CLAUDE.md の記述は「お願い」だが hook は強制 — ルール変更時は両方を更新する
- 同じ防御線: `bunfig.toml` の `minimumReleaseAge = 21`（公開 21 日未満のパッケージ版を拒否）+ `[install.security] scanner = "@bun-security-scanner/osv"`（install 時 OSV 脆弱性スキャン）
- **正規の実行経路は `make <target>` のみ**（実体は `docker-compose run --rm dev …`）

## Development Environment — Host Claude × devcontainer Runtime

Claude Code は host で起動し、bun 作業はすべて devcontainer 内（上の Security 参照）。入口は `make <target>`。

- **compose 定義は `.devcontainer/compose.yaml` をそのまま使う**（root に別の docker-compose.yml を作らない）。compose プロジェクト名は VS Code Dev Containers と同じ `marp_devcontainer` — イメージ・ネットワーク・起動中コンテナを共有し二重ビルドしない
- **`node_modules` はコンテナ用 Linux バイナリ**（biome / tsgo は darwin 非互換）。依存の再インストールは `make install`
- `make` は実行経路を自動判別: ① コンテナ内 → bun 直接実行（docker 不要） ② host + コンテナ起動中（VS Code attach or `make up` 済み） → `docker-compose exec`（高速） ③ 停止中 → `docker-compose run --rm`。連続作業は先に `make up`（`make down` は VS Code の devcontainer も止めるので注意）
- **hooks も同じルーティング**（`post-write.sh` は host 実行時に `bun` シムでコンテナへ委譲）。hook からコンテナに渡すパスは repo 相対パスにする（host の絶対パス `/Users/…` はコンテナに存在しない）
- Docker デーモンは colima。`Cannot connect to the Docker daemon` → `colima start`

## Commands

**Host では必ず `make`。コンテナ内シェル（`make shell` / VS Code devcontainer）では `bun run …` を直接実行してよい。**

| make (host) | コンテナ内の実体 | 内容 |
|---|---|---|
| `make install` | `bun ci` | 依存インストール（初回 / bun.lock 更新時） |
| `make format` / `make check` | `bun run format` / `check` | Biome 整形 / lint |
| `make typecheck` | `bun run typecheck` | tsgo（tsc ではない） |
| `make test` | `bun run test` | 全テスト（9 files / 4140 tests） |
| `make spellcheck` | `bun run spellcheck` | cspell |
| `make validate` / `make quality` / `make lint` | `bun run validate` / `validate:quality` / `lint` | schema 検証 / 品質チェック / 両方 |
| `make fix` / `make fix-all` | `bun run fix` / `fix:all` | 自動修正（fix:all = fix→data-uri→split→split:bullets→fix-svg→fix-svg-url-refs→index） |
| `make rebuild`（`-render` / `-export`） | `bun run rebuild` 等 | 全デッキ再構築（incremental・export は常に逐次） |
| `make single DECK=<partial>` | `bun run single <name> [render\|export\|all]` | 1 デッキだけ処理（部分一致） |
| `make stats` / `make doctor` / `make index` | `bun run stats` / `doctor` / `generate:index` | 品質統計 / ヘルスチェック / index.html 再生成 |
| `make bun ARGS="…"` / `make run CMD="…"` / `make shell` | 任意 bun / 任意コマンド / bash | 表にないもの全部 |

コンテナ内のみで使う操作:

```bash
bun run slides new "<topic>"              # 新デッキの雛形を docs/<ts>_<slug>/ に生成（config + シード JSON）
bun run slides render -c docs/<dir>/slides.config.yaml --in docs/<dir>/slides-data.json
bun run slides export -c docs/<dir>/slides.config.yaml -f html --in docs/<dir>/<name>.md   # -f html|pdf|pptx。特定デッキは必ず -c
bun run stats -- --verbose | --worst      # per-deck 内訳 / C・D グレードのみ
bun run rebuild -- --force                # キャッシュ無視の full rebuild
bun run dev [docs/<dir>]                  # watch mode（400ms debounce）
bun run team <session-id> <workspace> [impl-count] [review-count]   # Agent Teams / 監視: bun run team:status <id> [--watch]
```

### Testing

- `make test` = `bun test` — **9 test files / 4140 tests**。単一実行はコンテナ内で `bun test scripts/test-e2e.test.ts` / `bun test -t "normalizeSvg"`
- Tests live in **two** places: `src/__tests__/`（loader, markdown, slugify, svg, fit）と `scripts/test-*.test.ts` + `scripts/__tests__/quality.test.ts`
- `bun run check` = Biome（コード整形/lint）。`bun run lint` = スライド品質ゲート（validate + validate:quality）。**名前が紛らわしいが別物**

## Output Size Management — 32K Token Error Prevention

**2KB を超えうる出力はすべて Write tool でファイルへ書く（inline 出力禁止）。** スライド JSON・SVG・本ファイルの更新も同様。超過が続く場合は `CLAUDE_CODE_MAX_OUTPUT_TOKENS` を引き上げる。subagent の返り値にも適用 — 本体はファイルへ、返り値はメタ情報のみ。30 枚以上のデッキはチャンク分割生成（15 枚/chunk）→ `.claude/rules/slide-generation.md`。

## Interview-First Policy

タスク実行前に不明点（Goal / Scope / Output format / Constraints）を **1 問ずつ**確認し、明示的な承認（"OK" / "続けて" / "next"）を待つ。回答ごとに新たな未知が出れば 1 問だけ追質問。`/create-slides` は topic・audience・枚数・theme（gaia/default/uncover）・language（ja/en）も必須確認。

**ただし以下は確認せず即実行:** Shorthand 表に一致 / 明示的な slash command・skill の指名 / 答えが本ファイル・コード・会話の既決事項から確定する場合。

| User says | Interpretation |
|-----------|---------------|
| `"60more"`, `"60slides"`, `"100+"`, `"60枚"` | Target slide count = that number — start generating |
| `"全部並列"`, `"all at once"`, `"parallel"`, `"同時"` | Spawn all workers simultaneously in one message |
| `"続けて"`, `"continue"`, `"next"`, `"go"` | Proceed to next phase without asking |
| A number alone in slide context | Target slide count |
| `"all N topics"` / `"全N個"` | Create all N presentations in parallel |

> これらの Shorthand への聞き返しはセッション遅延を繰り返し起こしたため禁止。曖昧さのないコマンドとして扱う。

## Multi-Agent Development（デフォルト実行形態）

独立して並列化できる単位が **2 つ以上**あるタスクは、親が自分で作業せず**単位ごとに 1 subagent** を起動する。親は分解・起動・統合・報告のみ。詳細（親の責務表・subagent 契約・モデル振り分け・実行パラメータ）→ `.claude/rules/agent-teams.md`。

- 全 subagent を **1 メッセージ内で同時に** `Agent` 呼び出し（逐次起動が最頻の失敗）。10+ workers は 5-7 個のウェーブに分割
- 各ワーカーに**重複しない出力パス**（`slides-data-part{N}.json`）。返り値は成果物本体ではなく**メタ情報のみ**（パス + 枚数 + 品質）
- render / export は**常に親が逐次実行**（Marp CLI キャッシュ競合）
- モデル振り分けの原則: スライド生成 = `opus`、実装・調査 = `sonnet`、機械実行 = `haiku`、親 = `inherit`。`bun run validate` / `stats` で機械判定できる作業は haiku で足りる
- 1 ファイルの小さな修正・確認だけなら subagent を使わず親が直接やる

## Slide Generation（要点）

Pipeline（skip 禁止）: `outline → config YAML → JSON data (Write tool) → validate → render → export HTML`

- フィールド名は `content`（**`bullets` ではない**）。`layout` は `default | center | section` のみ → `.claude/rules/slide-generation.md`
- SVG: **`url(#id)` 全面禁止**・root は `viewBox` のみ（width / height / vh 禁止）→ `.claude/rules/svg.md`
- 品質基準: assertive title 60%+ / SVG 比率 50%+ / BLUF（スライド 2 に結論）/ SCQA → `.claude/rules/slide-design.md`
- config は `marp:` ネスト必須・`output.dir` はフルパス → `.claude/rules/validation.md` / `output-structure.md`
- Themes: `gaia`（tech/LT）・`gaia`+`class: invert`（dark）・`default`（business）・`uncover`（minimal）

## Architecture Pitfalls（非自明のみ — モジュール表は `.claude/rules/architecture.md`）

- Front-matter と最初のスライドは `\n\n` のみで連結。`\n\n---\n\n` はスライド間のみ、front-matter 直後には使わない
- `slides-data.json` を書き換えるスクリプトは `src/model/presentation.ts`（Zod 検証 + atomic write）を使う。読み取り専用の集計は `scripts/lib/presentation-loader.ts`（高速・無検証）
- **root に `marp.config.mjs` を置かない**（export は `--no-config` で spawn され無視される）。export は 1 デッキ 120s timeout
- `output.dir` は Zod `.transform()` で絶対パス化される — 相対値は cwd 基準で silent に解決されるため必ず `docs/<timestamp>_<slug>` のフルパス
- Grade formula: `svgRatio×40 + assertiveRatio×40 + subtitleRatio×20`（A ≥ 70）。assertive/subtitle は `layout: "default"` のみ対象
- `docs/index.html` は `scripts/index-template/`（shelf.css / shelf.js）から生成 — 生成物ではなくテンプレートを編集。PDF は repo-root `pdf/` に置くと `generate:index` が `docs/pdf/` にミラー

## Conventions

- Bun idioms: `Bun.file()` / `Bun.write()` / `Bun.spawn()`。互換モジュールは `node:` prefix
- Imports: `.ts` ファイルにも `.js` 拡張子（ESM bundler）。Biome が import を自動整列
- Formatter: Biome（**tab indent**）。Type checker: tsgo（`@typescript/native-preview`）
- Source code / comments は英語。スライド出力の既定は日本語（`language: "ja"`）
- 依存追加はコンテナ内で `make bun ARGS="add <pkg>"`。bunfig.toml が公開 21 日未満の版を拒否するため、追加直後の最新版が入らないのは仕様
